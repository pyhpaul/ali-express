# AliExpress 分页总量驱动设计

日期：2026-05-09

## 目标

把当前分页逻辑从“每页抓固定数量”改成“按总量目标自动翻页”，使 `--max-items` 表示本次抓取的总商品上限，而分页只是为了在当前页不够时继续补量。

## 现状证据

当前实现存在以下行为：

- `collect_raw_products()` 按 `pages` 循环逐页抓取。
- `_collect_current_page()` 在每一页内部都按 `max_items` 提前截断。
- 因此 `--max-items 20 --pages 2` 的实际效果是“第 1 页抓 20 条 + 第 2 页抓 20 条”，不是“总共抓 20 条，必要时最多翻 2 页”。

这和用户对分页的直觉不一致。用户视角下，分页的前提应当是：先尽量拿完当前页数据，当前页不够时再翻下一页。

## 方案

### 参数语义

- `--max-items`：本次抓取的总商品上限。
- `--pages`：可选的最大访问页数上限。
  - 指定时：最多访问该页数。
  - 未指定时：根据 `--max-items` 自动翻页，直到累计唯一商品数达到目标，或无法继续翻页。

### 采集行为

- 单页采集函数只负责尽量收集当前页已经加载出的全部商品，不再按 `max_items` 做页内截断。
- 外层流程负责：
  - 跨页去重
  - 判断累计唯一商品数是否已达到 `max_items`
  - 判断是否还能继续翻页
  - 在最终汇总阶段统一按 `max_items` 做一次截断

### 停止条件

抓取在以下任一条件满足时停止：

1. 累计唯一商品数达到 `max_items`
2. 已达到 `--pages` 指定的页数上限
3. 页面不存在下一页控件，或控件不可操作
4. 触发翻页后商品签名未变化，判定翻页失败

## 边界

- 不引入新的 CLI 参数。
- 不改变现有 CSV 字段结构。
- 不强制抓满指定页数；即使设置了 `--pages`，只要已达到 `max_items` 也应提前停止。
- 若单页商品数已经超过 `max_items`，最终输出仍只保留前 `max_items` 个去重结果。

## 兼容性影响

这是一个有意的行为调整：

- 旧行为：`--pages` 默认值为 `1`，未指定时通常只抓单页。
- 新行为：未指定 `--pages` 时，系统会按 `max_items` 自动翻页。

这会让默认抓取时间变长，但更符合用户对“目标总量”的理解。若用户只希望单页抓取，应显式指定 `--pages 1`。

README 和命令帮助需要同步更新，明确：

- `--max-items` 是总量目标
- `--pages` 是可选页数上限
- 未指定 `--pages` 时会自动翻页

## 验证

- `tests/test_cli.py`
  - `--pages` 未传时解析为 `None`
  - `--pages 3` 时解析为 `3`
  - `--pages 0` / 负数时仍报错
- `tests/test_browser.py`
  - 单页采集函数不再按 `max_items` 截断
  - `collect_raw_products()` 在 `pages=None` 时会自动翻页，直到累计唯一商品数达到 `max_items`
  - 指定 `pages=N` 时，最多访问 `N` 页
  - 最终只在汇总阶段按总量做一次截断
  - 翻页失败或无下一页时能正确停止
- 手工验证
  - 使用当前目录的 `.browser-profile`
  - 运行 `--max-items 20`，验证会自动翻页并在够数时停止
  - 运行 `--max-items 80 --pages 2`，验证最多抓取两页
